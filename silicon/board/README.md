# PYNQ-Z2 bring-up: mac_nxn_array

What to copy to the board, and where, to run `run_mac_lns.ipynb` against the
synthesized `mac_nxn_array` kernel.

## What to copy

From the Vitis/Vivado export (built separately, not part of this repo tree):
- `mac_lns.bit`
- `mac_lns.hwh`

From this repo:
- `silicon/board/run_mac_lns.ipynb`
- `silicon/board/vectors/` (generated by `silicon/host/build_and_run.sh` --
  30 `case*.bin` files + `manifest.json`)

## Where

On the PYNQ-Z2, put all four items in the same Jupyter working directory,
e.g.:

```
/home/xilinx/jupyter_notebooks/mac_lns/
├── mac_lns.bit
├── mac_lns.hwh
├── run_mac_lns.ipynb
└── vectors/
    ├── manifest.json
    ├── case0_a.bin
    ├── case0_b.bin
    ├── case0_expected.bin
    ├── ...
    └── case9_expected.bin
```

`run_mac_lns.ipynb` loads `mac_lns.bit` with a relative path and reads
`vectors/` relative to its own directory, so this flat layout is required --
don't nest the notebook or the vectors folder any deeper.

## The one landmine: `.hwh` must sit next to the `.bit`

`Overlay('mac_lns.bit')` looks for `mac_lns.hwh` **in the same directory,
with the same basename** (only the extension differs). PYNQ uses the `.hwh`
to build the IP dictionary (`ol.ip_dict`, register maps, `ol.<ip_name>`
attribute access) -- without it next to the bitstream, the overlay loads but
IP discovery breaks or silently returns the wrong register map. If the
export tool named it something else (e.g. `design_1_wrapper.hwh`), rename it
to match the `.bit` basename before copying it over.

## Regenerating vectors

Vectors are produced on the host (not the board) by:

```
bash silicon/host/build_and_run.sh   # from the repo root
```

which (re)builds `silicon/host/gen_vectors` against the exact synthesized
kernel source (`silicon/src/mac_silicon.cpp`) and the shared mul/add units,
then runs it to (re)populate `silicon/board/vectors/`. Re-run it any time the
kernel source or the vector seeds change; then re-copy `vectors/` to the
board.
